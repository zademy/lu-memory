= Contributing to lu-memory
:toc: macro
:toc-title: Table of Contents

First off, thank you for considering contributing to `lu-memory`! This project is an MCP (Model Context Protocol) Server for long-term memory management using SQLite, authored and maintained by **zademy**. It's people like you that make it such a great tool. We welcome contributions from everyone.

toc::[]

== Code of Conduct
By participating and contributing to this project, you agree to abide by our link:CODE_OF_CONDUCT.md[Code of Conduct]. We expect all contributors to maintain a professional, respectful, and welcoming environment within the **zademy** community.

== Getting Started
To get started with development, you will need the following tools installed on your system:

* Java Development Kit (JDK) 25 or higher
* Apache Maven (or use the provided `mvnw` wrapper)
* SQLite (the JDBC driver will handle local db creation automatically)

=== Setting up the Development Environment
1. Fork the repository and clone it to your local machine:
[source,bash]
----
git clone https://github.com/zademy/lu-memory.git
----
2. Navigate to the project directory:
[source,bash]
----
cd lu-memory
----
3. Build the project using the Maven wrapper:
[source,bash]
----
./mvnw clean install
----

== How to Contribute

=== Reporting Bugs
If you find a bug, please create an Issue in the GitHub repository. Provide as much related information as possible:
* Clear steps to reproduce the bug.
* The expected outcome versus the actual behavior.
* Environment details (OS, JDK version, memory setup, etc.).
* Any relevant stack traces, logs, or screenshots.

=== Suggesting Enhancements
We are always looking for ways to improve the `lu-memory` capabilities. If you have a feature request or an enhancement idea, please open an Issue describing:
* The problem or limitation you have encountered.
* Your proposed solution or new feature.
* Real-world use cases where this feature would be helpful.
* Any alternative approaches you might have considered.

=== Submitting Pull Requests
We gladly accept Pull Requests (PRs). When submitting one:

1. **Create a new branch** from `main` for your feature or bug fix:
[source,bash]
----
git checkout -b feature/my-awesome-feature
----
or
[source,bash]
----
git checkout -b fix/issue-123
----
2. **Implement your changes**, ensuring the project builds successfully without any warnings.
3. **Write tests** to cover your new functionality or prove that the bug was fixed.
4. **Commit your changes** using clear and descriptive commit messages.
5. **Push** your branch to your fork.
6. Open a **Pull Request** against the up-stream `main` branch. Provide a comprehensive description of the changes inside the PR template.

== Architecture & Coding Guidelines
As a contributor, please keep in mind the following principles when modifying the source code:

* **Java Conventions**: Follow standard Java coding conventions.
* **SOLID Principles**: Keep components modular, testable, and maintainable. Use interfaces to define contracts where appropriate.
* **Clean Code**: Use meaningful self-documenting names for variables, methods, and classes. Keep methods focused on doing one thing well.
* **Documentation**: Document any complex logic, and update relevant Markdown / AsciiDoc files if introducing new commands or workflows.

== Testing
We use JUnit and Spring Boot Testing utilities. Please make sure that all existing tests pass before pushing:
[source,bash]
----
./mvnw test
----
If you are adding an entirely new MCP tool, consider adding unit/integration tests to verify that the tool handles edge cases and interacts correctly with the underlying SQLite (`lu-memory.db`) storage.

== Database Changes
If your contribution modifies the core memory schema (e.g., editing `create-tables.sql`), ensure that:
* The FTS5 (Full-Text Search) mechanisms are left intact or updated accurately.
* Backwards compatibility is preserved, or explicit migration guidance is provided.

Thank you again for contributing to `lu-memory`! Your efforts help make this project better for everyone at **zademy**.
